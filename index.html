AI-Assisted Workflow
Linear & Claude at Arkyn

A practical walkthrough of how we use Linear for project tracking and Claude as an AI co-pilot across product, development, and QA — for the FastWork iOS, Android, and MW & SAP teams.

The Big Picture

Our workflow connects four systems into a single feedback loop:

Zendesk (customer reports)
→
Linear (plan & track)
→
GitHub (build & review)
→
Sentry (monitor)

Claude sits alongside each stage — helping product managers draft specs in Cowork, helping developers implement and review code with Claude Code, and helping testers verify behavior with Claude in Chrome. The goal is to reduce context-switching: information flows from one tool to the next without manual copy-paste.

Product & Planning Workflow

Product & PM

1. Capture the requirement

When a feature request or bug arrives — from a customer call, a Zendesk ticket, or an internal idea — create a Linear issue in the appropriate team. Use Cowork (Claude's desktop tool) to help draft the issue description: paste in raw notes, a Zendesk transcript, or a Sentry stacktrace and ask Claude to structure it into a clear problem statement, acceptance criteria, and test plan.

Raw notes / Zendesk ticket
→
Cowork: "Structure this into a Linear issue"
→
Linear issue created
Product & PM

2. Write a feature spec

For non-trivial features, we write a spec before implementation begins. The spec lives in the repo at docs/specs/<feature-name>.md and covers: goal, data models, UI description, reference feature, design decisions, acceptance criteria, test plan, and implementation steps.

Cowork can draft the spec for you. Give it the Linear issue, any design mockups, and mention a reference feature ("follow the pattern in TimeConfirmations") — it will produce a first draft you can review and refine.

Product & PM

3. Break into tasks

Once the spec is approved, break it into Linear sub-issues assigned to the right team. Features that span platforms get a parent issue (or project) with sub-issues in both FWIOS and FWAND. This is where cross-platform handoff begins — more on that below.

Development Workflow

Developer

1. Pick up an issue

Assign yourself, move it to In Progress. Create a branch using Linear's naming convention — Linear auto-links branches that contain the issue ID (e.g. tjuela/fwios-142-lazy-measurement-points).

Developer

2. Implement with Claude Code

Claude Code runs in your terminal and has full access to the repo. It reads CLAUDE.md for project conventions (MVVM, @Observable view models, GRDB query patterns, the WorkFeatures package structure) so it writes code that fits our architecture from the start.

Typical workflow:

  1. Point Claude Code at the spec: "Read docs/specs/time-confirmations-overview.md and implement step 1."
  2. Claude Code creates the view model, view, and previews following the MVVM pattern.
  3. You review, iterate, and ask Claude Code to write tests (Testing framework, not XCTest).
  4. When done, ask Claude Code to create the PR — it auto-detects the Linear issue ID from the branch name and adds Closes FWIOS-142 to the description.
Tip: Claude Code's memory system learns your preferences over time. If you tell it "don't mock the database in integration tests," it remembers that for future sessions.
Developer

3. PR & review

PRs follow a consistent structure: summary, Linear issue reference (Closes FWIOS-142), ADO reference if applicable (Fixes AB#12345), and test plan. Claude Code generates this automatically. Reviewers can use Claude Code to understand unfamiliar code — "Explain what this GRDB association query does" — without leaving the terminal.

Cross-Platform Handoff

One of Linear's biggest strengths for us is enabling one platform to define work for another. When the iOS team ships a feature, the implementation details become a blueprint for Android (and vice versa).

Cross-Platform

How it works in practice

Example: Operation Quantity — this feature was built on iOS first. The iOS implementation defined exactly what data to display (OrderOperation.work + OrderOperation.workUnitOfMeasure) and how. Once iOS shipped:

  1. A Linear project (Operation Quantity) was created spanning both teams.
  2. The iOS spec and implementation became the reference for the FWAND issue.
  3. The Android developer can point Claude Code at the iOS spec: "Read the Operation Quantity spec. Implement the equivalent in our Kotlin/Compose codebase."
  4. Claude Code adapts the patterns to Android idioms while preserving the exact same business logic and UI behavior.
Tip: Use Linear project descriptions to document platform-agnostic behavior (what the feature does), and team-specific sub-issues for implementation details (how each platform builds it). This way the spec is shared but the tasks are independent.
Product & PM

Defining tasks for the other platform

Product managers and lead devs can use Cowork to generate well-structured cross-platform issues. For example:

  1. Open Cowork with access to the repo.
  2. "The Date Filter for Data Synchronization feature is complete on iOS. Read the spec at docs/specs/date-filter.md and create a Linear issue for the Android team describing what needs to be built, referencing the iOS implementation as a guide."
  3. Cowork drafts the issue with the right level of detail — UI behavior, data flow, edge cases — so the Android team has a clear brief without needing to reverse-engineer the iOS code.

Testing & QA Workflow

Tester

1. Manual testing with Claude in Chrome

Testers can use Claude in Chrome to interact with web-based test management tools, Zendesk, and Sentry dashboards. Ask Claude to summarize a Sentry issue's stacktrace, cross-reference it with the Linear issue, or draft a bug report from what you're seeing in the browser.

Tester

2. Automated UI testing

For iOS, we use agent-device to drive the simulator via the accessibility tree. Claude Code can write and run UI tests interactively: 

# Open the app on the simulator
agent-device open io.arkyn.development.work

# Inspect the current screen
agent-device snapshot -i

# Tap an element and verify the result
agent-device press 'label="sidebar.item.myWorkOrders"'
agent-device wait visible 'label="Order List"' 3000
agent-device screenshot ./proof.png

All accessibility identifiers are centralized in FastWork/AccessibilityIdentifiers.swift using dot-separated namespacing (e.g. sidebar.item.settings). This keeps tests stable across localization and UI changes.

Cowork can help testers draft structured bug reports: paste in a screenshot and raw notes, and Claude structures it with reproduction steps, expected vs. actual behavior, and suggests which team/project to file it under.

Sentry Integration

Integration

Crash monitoring → Linear issues

Sentry captures crashes and errors from the production app. Our workflow connects Sentry to Linear so issues flow automatically:

  1. Sentry alert fires — a new crash or error threshold is breached.
  2. Linear issue created — Sentry's Linear integration auto-creates an issue in the FastWork Support project with the stacktrace, affected users count, and device info.
  3. Triage — the issue is reviewed and moved to the appropriate team (FWIOS or FWAND).
  4. Investigation with Claude — developers can use Claude Code to analyze the Sentry stacktrace in context: "This Sentry crash points to OrderDetailsWrapperViewModel. Look at that file and explain what could cause this nil unwrap."
Tip: Cowork has direct access to Sentry via its MCP connector. You can ask: "Show me the top unresolved Sentry issues for FastWork this week" — and it will query Sentry, summarize the findings, and help you create Linear issues for the critical ones.

Zendesk Integration

Integration

Customer tickets → Linear issues

When a customer reports a problem through Zendesk, the support team can escalate it to engineering by creating a Linear issue in the FastWork Support project. The process:

  1. Zendesk ticket arrives — support agent reviews and determines it needs engineering attention.
  2. Create Linear issue — using Zendesk's Linear integration or manually. Include the ticket link and key details.
  3. Cowork assists — if the support agent uses Cowork, they can paste the Zendesk conversation and ask Claude to extract the technical details, identify which area of the app is affected, and draft a well-structured Linear issue.
  4. Engineering triages — the issue is moved from FastWork Support to the appropriate team and prioritized.

Once the fix is shipped, the Linear issue status update flows back to the Zendesk ticket, closing the loop with the customer.

GitHub Integration

Integration

Automatic linking

Linear and GitHub are tightly integrated:

  1. Branch names containing the issue ID (e.g. tjuela/fwios-142-...) auto-link the branch to the Linear issue.
  2. PR descriptions with Closes FWIOS-142 automatically transition the Linear issue to "Done" when the PR merges.
  3. Azure DevOps — if a Linear issue references an ADO task, include Fixes AB#12345 in the PR description to auto-close the ADO work item too.

Claude Code handles all of this automatically when creating PRs — it reads the branch name, finds the Linear issue ID, and structures the PR description correctly.

Claude Code — For Developers

Developer Tool

What it is

Claude Code is a terminal-based AI assistant that has full access to your codebase. It reads, writes, and executes code. It understands our project conventions via CLAUDE.md and remembers your personal preferences across sessions.

Key capabilities

  1. Implement from specs — point it at a feature spec and it builds the view model, view, previews, and tests following our MVVM + GRDB patterns.
  2. Create PRs — auto-detects Linear issue IDs, structures the PR description, includes ADO references.
  3. Debug with context — paste a Sentry stacktrace and ask it to investigate the root cause in the actual codebase.
  4. Write tests — generates tests using Apple's Testing framework (not XCTest) with proper mocks via ActionService.setup.
  5. Drive the simulator — uses agent-device to interact with the app, take screenshots, and verify UI behavior.
  6. Search Linear — can query Linear issues, projects, and teams to understand context before starting work.

Cowork — For Product & PM

PM Tool

What it is

Cowork is Claude's desktop application for non-developers. It connects to Linear, Sentry, Zendesk, Slack, GitHub, and your local files — giving product managers and project leads an AI assistant that can read and act across all your tools.

Key capabilities

  1. Draft Linear issues — from raw notes, Zendesk transcripts, or Sentry stacktraces.
  2. Write feature specs — generates structured specs using our template and existing code as reference.
  3. Cross-platform task creation — reads an iOS implementation and creates a structured brief for the Android team (or vice versa).
  4. Status updates — ask Cowork to summarize progress on a Linear project, pulling data from issues, PRs, and milestones.
  5. Create presentations and docs — generate slide decks, reports, and documents from project data.

Claude in Chrome — For Testers & Support

Tester & Support Tool

What it is

Claude in Chrome is a browser extension that lets Claude see and interact with the page you're viewing. Useful for testers working in web-based tools like Sentry, Zendesk, or internal dashboards.

Key capabilities

  1. Summarize Sentry issues — navigate to a Sentry issue page and ask Claude to explain the crash in plain language.
  2. Draft bug reports — while looking at unexpected behavior in a web app, ask Claude to write up a structured bug report.
  3. Navigate Zendesk — help support agents extract technical details from customer conversations.
  4. Compare behavior — take screenshots of expected vs. actual behavior and ask Claude to identify differences.

Conventions & Tips

Linear conventions

  1. Issue titles should be concise and action-oriented: "Add date filter to sync settings" not "Date filter feature".
  2. Labels — use labels to categorize: bug, feature, improvement, support.
  3. Projects group related work that spans multiple sprints. Keep project descriptions up to date — they serve as the single source of truth for what the feature does.
  4. Sub-issues break a project into implementable chunks. Assign each to a specific team.
  5. Cycles (sprints) — assign issues to the current cycle for sprint planning. Keep cycle scope realistic.

Claude conventions

  1. Be specific — "Read the spec at docs/specs/time-confirmations-overview.md and implement step 3" beats "add time confirmations."
  2. Reference existing patterns — "Follow the pattern in StationDetailsView" gives Claude a concrete example to mirror.
  3. Let Claude read first — don't paste code into the chat. Let Claude Code read the files directly so it has full context.
  4. Review what it writes — Claude is a co-pilot, not autopilot. Always review generated code, specs, and issues before they go live.
  5. Give feedback — if Claude gets something wrong, tell it. It learns your preferences across sessions via its memory system.

Branch & PR naming

# Branch: include the Linear issue ID
git checkout -b tjuela/fwios-142-lazy-measurement-points

# PR description: Linear auto-links and transitions

Fixes AB#12345  # if ADO task exists
Note: Claude Code generates PR descriptions automatically following these conventions. You just need to make sure your branch name includes the issue ID.
