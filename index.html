Linear & Claude at Arkyn

A complete walkthrough of our process for shipping software with Linear and Claude.

The Big Picture

The Arkyn team uses Linear to track all work and Claude (through Claude Code and Cowork) to execute that work. The workflow integrates with Sentry, GitHub, Slack, and Scheduled Tasks to create a seamless system where issues become projects, projects become code, and code gets deployed.

Core principle: Let Claude read the issue in Linear, understand context from tools, and ship code without context-switching.

Our Linear Teams

Work is organized by product and platform. Each product has a cross-cutting team for planning and coordination, plus platform-specific teams for implementation:

How Teams Work Together

Cross-platform work lives as a parent Story in the cross-cutting team (e.g. FastWork) with sub-issues in each platform team. This way the parent tracks overall progress while each platform team sees their piece on their own board.

Setup Overview

To get the full workflow running, you need:

  1. Claude Code — The IDE for working with projects
  2. Cowork — The plugin ecosystem for integrations
  3. MCP Integrations — Tools like Slack, GitHub, Sentry, and more

Claude Code Setup

Claude Code is Anthropic's official CLI for Claude. It lets you open projects and work with them directly.

Installation:

npm install -g @anthropic-ai/claude-code

Usage:

claude-code open /path/to/project

Once opened, you can ask Claude to read files, run commands, make commits, and push code—all without leaving the tool.

Cowork Setup

Cowork is the plugin platform that extends Claude's capabilities. Plugins provide access to tools like Linear, Slack, GitHub, and more.

Install a plugin:

claude setup-cowork

This opens a guided setup flow where you can:

  • Choose your role (designer, engineer, product manager, etc.)
  • Select matching plugins
  • Authenticate with each tool
  • Test the connection

Once set up, your plugins persist and are available in all future Claude sessions.

The Product Brief

A good issue includes:

What

What needs to be built? A one-sentence summary.

Why

Why is it important? Customer impact, OKR alignment, or business metric.

How

How might we do it? Acceptance criteria, rough approach, design specs.

Who

Who's involved? Platform badges, owner, blocked by, dependencies.

What Claude Does

Dev

Reads the issue and understands the requirements.

Searches the codebase to find similar patterns and understand architecture.

Writes code in the correct style and format.

Runs tests to validate the implementation.

Commits and pushes to GitHub, with PR descriptions auto-generated from the branch name.

Branch Naming Convention

Include the Linear issue ID in the branch name:

git checkout -b DEV-123-add-user-auth

Pro Tip: The issue ID in the branch name helps Linear and GitHub stay in sync. When you push a branch, the PR is linked automatically, and when you merge, the issue closes.

Code Review

Claude can request a Copilot code review on your PR:

claude-code request-copilot-review [owner/repo PR#]

Copilot provides feedback on code quality, performance, and style. After addressing feedback, Claude pushes another commit.

Dependencies

  • iOS issue is blockedBy backend API issue
  • When backend issue closes, Linear notifies the iOS team
  • Claude can read dependencies and suggest parallel work

What Claude Does

  • Fetches error details from Sentry
  • Runs root cause analysis (using Seer)
  • Creates a Linear issue with stack trace and reproduction steps
  • Assigns to the relevant team

GitHub

Dev

GitHub is where code lives. Claude pushes commits, creates PRs, and manages branches.

What Claude Does

  • Reads existing code to understand patterns
  • Creates branches with the Linear issue ID
  • Writes code and commits with clear messages
  • Opens PRs with auto-generated descriptions
  • Responds to review feedback and updates the branch

Pro Tip: Branch names like DEV-123-add-auth automatically link the PR to the Linear issue. GitHub and Linear stay in sync, and when you merge, the issue closes automatically.

Slack

PM

Slack is how the team communicates. Claude can read channels, post updates, and notify teams.

What Claude Does

  • Reads Slack threads to understand context
  • Posts status updates to channels
  • Notifies teams when issues are created or updated
  • Responds to messages in threads
  • Creates channels for projects

Note: Claude respects permissions and only posts to channels where it has access.

Scheduled Tasks

Ops

Scheduled Tasks let Claude run periodic workflows automatically—like daily standups, weekly reports, or nightly builds.

What Claude Does

  • Runs tasks on a schedule (daily, weekly, monthly, or custom)
  • Executes one-time tasks at a specific time
  • Reads Linear issues and creates status reports
  • Sends Slack notifications
  • Generates metrics and dashboards

Example: Daily Standup

Schedule a task to run every morning at 9 AM:

claude-code schedule "Daily standup" "0 9 * * *"

Claude reads the team's Linear cycle, gathers today's tasks, and posts a summary to Slack.

Pro Tip: Use Scheduled Tasks to automate routine work like status checks, metrics collection, or release notes generation.

CLAUDE.md for Linear

Add the following to your project's CLAUDE.md file to give Claude Code full context on how to work with your Linear workspace. This ensures Claude follows your labeling conventions, sub-issue patterns, and PR linking rules.

Pro Tip: Put this in your global ~/.claude/CLAUDE.md to apply it across all repositories, or in a project-level CLAUDE.md for project-specific customization.

Important: Clone all relevant repositories locally so Claude can read code across projects when working on cross-platform Linear issues. For example, when creating sub-issues for iOS and Android from a parent Story, Claude can reference the existing implementation in each codebase to write accurate issue descriptions.
## Local Repositories

All repositories should be cloned under a shared parent directory:

| Repository | Description |
|---|---|
| `your-ios-app` | iOS/macOS app |
| `your-android-app` | Android app |
| `your-shared-library` | Shared utilities or packages |
| ... | Add all repositories relevant to your team |

When working on a Linear issue that spans multiple repositories, read the relevant
source code to understand the current implementation before creating sub-issues or
writing code. Cross-platform issues often require understanding both the iOS and
Android codebases to write accurate platform-specific sub-issue descriptions.

## Linear

### Workspace Structure
- **Teams**: FastWork (cross-cutting), FastWork iOS (FWIOS), FastWork Android (FWAND), FastWork MW & SAP (FWMW)
- **Initiatives**: Field Knowledge Platform, Platform Reliability, Customer Success, Core Product, Getting Started
- **Projects** are labeled: Epic (feature work), Hypercare (post-go-live support), Infrastructure (platform/tooling)

### Issue Labels
Apply exactly one type label per issue:
- **Bug** — Something broken or not matching expected behavior
- **Feature** — Net-new capability that didn't exist before
- **Improvement** — Enhancement to existing functionality (UX polish, performance, validation)
- **Story** — User-facing requirement, typically a parent with sub-issues
- **Task** — Internal/technical work with no direct user-facing change
- **Technical Debt** — Cleanup or modernization not tied to a feature or bug

### Sub-Issues
Use sub-issues for cross-platform work and breaking down larger items:

**Cross-platform pattern:**
1. Create a parent Story in the FastWork team with the overall description
2. Create sub-issues in each platform team (FWIOS, FWAND, FWMW) with platform-specific details
3. The parent tracks overall progress; sub-issues are assigned, estimated, and cycled independently

Example:
```
FAS-37 [Story] Configurable operation cell: TO display
  ├── FWIOS-215 [Feature] iOS implementation
  ├── FWAND-24  [Feature] Android implementation
  └── FWMW-2   [Task]    MW configuration
```

**Story breakdown:** Split Stories into Task/Feature sub-issues per implementation step.

**Do NOT create sub-issues** for simple bugs, single-platform tasks, or work under one day.

### Creating Issues
- Always set the appropriate type label (Bug, Feature, Improvement, Story, Task, Technical Debt)
- Assign to the correct team based on platform scope
- Set project if the work belongs to an existing project/epic
- Use `blockedBy`/`blocks` when work depends on another issue
- When creating cross-platform work, always create a parent Story in the FastWork team first

### Searching for Context
Before starting work on an issue:
- Read the Linear issue for requirements and acceptance criteria
- Check for related issues, sub-issues, and blocking dependencies
- Look at the project description for broader context
- Check if there are customer needs or ADO references linked to the issue

### Branch Naming
Linear auto-generates branch names with the issue ID (e.g. `juel/fwios-213-show-attachments`).
Always use these for automatic PR-to-issue linking.

### PR References
- Linear issues are auto-linked via branch name — no manual action needed
- For Azure DevOps work items: add `AB#12345` in the PR description to auto-transition
- Add `Closes FWIOS-123` in the PR description to auto-transition the Linear issue on merge
- When a Linear issue references an ADO task, include both references in the PR
